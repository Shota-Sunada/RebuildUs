# RebuildUs 開発ガイド: Roles & Modifiers 実装仕様書

## 1. 基本アーキテクチャ
RebuildUs は、Among Us の標準的な役職システムを拡張し、独自の役職と属性（Modifier）をインスタンスベースで管理しています。

### クラス構成
- **Role系**: `PlayerRole` を継承します。通常は `MultiRoleBase<T>` または `SingleRoleBase<T>` を使用します。
- **Modifier系**: `ModifierBase<T>`（または `PlayerModifier`）を継承します。これらは、メインの役職に付与される「追加の特性（例：恋人、マッドメイトなど）」です。

## 2. 新しい役職 (Role) の作成手順

### 1. Enum への登録
`RebuildUs/Enums/RoleType.cs` に新しい ID を追加します。

### 2. クラスの作成
`RebuildUs/Roles/[Team]/` 内に新しい `.cs` ファイルを作成します。

```csharp
[RegisterRole(RoleType.MyNewRole, RoleTeam.Crewmate, typeof(MultiRoleBase<MyNewRole>), nameof(CustomOptionHolder.MyNewRoleSpawnRate))]
internal class MyNewRole : MultiRoleBase<MyNewRole>
{
    public MyNewRole()
    {
        StaticRoleType = CurrentRoleType = RoleType.MyNewRole;
    }
}
```

### 3. 利用可能なカスタムイベント
`[CustomEvent(CustomEventType.[Type])]` をメソッドに付与することで、特定のタイミングでコードを実行できます。実行対象が「自分自身のみ」か「役職保持者全員」かに注意してください。

| イベント名 (`CustomEventType`) | 実行対象 | 引数 | 実行タイミング / 説明 |
| :--- | :--- | :--- | :--- |
| **FixedUpdate** | **自分のみ** | なし | 毎フレーム実行されます。クールダウンの減少や状態チェックに使用します。 |
| **OnKill** | **自分のみ** | `PlayerControl target` | 自分が誰かをキルした時に実行されます。`target` はキルされたプレイヤーです。 |
| **OnDeath** | **自分のみ** | `PlayerControl killer` | 自分が死亡した時に実行されます。`killer` は自分を殺したプレイヤーです。 |
| **OnMeetingStart** | **全員** | なし | 会議が開始された時に、この役職を持つ生存者全員で実行されます。 |
| **OnMeetingEnd** | **全員** | なし | 会議が終了した時に、この役職を持つ生存者全員で実行されます。 |
| **OnIntroEnd** | **全員** | なし | ゲーム開始時の役職紹介演出が終了した時に実行されます。 |
| **HandleDisconnect** | **全員** | `PlayerControl player`, `DisconnectReasons reason` | 誰かが切断した時に実行されます。`player` は切断したプレイヤーです。 |
| **OnFinishShipStatusBegin** | **自分のみ** | なし | マップ（ShipStatus）の生成・開始処理が完了した直後に実行されます。 |

## 4. 非同期処理と遅延実行 (Coroutines)

役職の能力で「数秒後に実行する」といった遅延が必要な場合、`FixedUpdate` でタイマーを自前で実装するよりも、Unity の **Coroutine** を使用する方が効率的です。

### 実装例: 遅延通報
`Bait` のように、死亡した数秒後にアクションを起こす場合のパターンです。

```csharp
[CustomEvent(CustomEventType.OnDeath)]
internal void OnDeath(PlayerControl killer)
{
    if (killer == null || Player != PlayerControl.LocalPlayer) return;

    // HudManager などの生存している MonoBehaviour を利用してコルーチンを開始
    FastDestroyableSingleton<HudManager>.Instance.StartCoroutine(DelayedAction(killer));
}

private System.Collections.IEnumerator DelayedAction(PlayerControl killer)
{
    yield return new WaitForSeconds(delayTime);

    // 実行前の生存チェック等
    if (Helpers.IsGameOver) yield break;

    // ロジック実行...
}
```

> **注意**: コルーチンを使用する場合は、`System.Collections.IEnumerator` を戻り値にします。

## 5. 新しい属性 (Modifier) の作成手順

### 1. Enum への登録
`RebuildUs/Enums/ModifierType.cs` に新しい ID を追加します。

### 2. クラスの作成
`RebuildUs/Roles/Modifier/` 内に作成します。Modifier は役職とは別に複数持つことが可能です。

```csharp
internal class MyModifier : ModifierBase<MyModifier>
{
    // Modifier 特有のロジック
}
```

## 4. 実装における重要なルール (Banned APIs)

IL2CPP 環境でのパフォーマンスと安定性を守るため、以下のルールを厳守してください。

- **`CastFast<T>()` の使用**:
  通常の `Cast<T>()` ではなく `Il2CppHelpers` にある `CastFast<T>()` を使ってください（リフレクションのオーバーヘッド削減）。
- **`FastDestroyableSingleton<T>.Instance`**:
  `HudManager` や `PlayerControl` などのシングルトンへのアクセスは必ずこちらを使用します。
- **LINQ の禁止**:
  `Update` や `FixedUpdate`、頻繁に呼ばれるパッチ内での `System.Linq` は禁止です。`foreach` や `for` を使い、リストには `GetFastEnumerator()` を併用してください。
- **文字列操作**:
  ホットパス（Update等）での文字列連結 (`+` や `$"{}"`) は控え、`StringBuilder` を活用してください。

## 5. ネットワーク（RPC）
役職固有の能力で同期が必要な場合は、`RPCSender` を使用します。

```csharp
using RPCSender sender = new(Player.NetId, CustomRPC.MyCustomAbility);
sender.Write(targetPlayerId);
// 破棄 (Dispose) を忘れないように using を使用
```

## 6. ローカライズ
ハードコードは避け、必ず `TrKey.[KeyName]` を使用してください。追加後は、今回作成した `AddTranslation.sh` (または `.bat`) を実行して、JSON と Enum を更新します。
